# 新手教程 — 用大白话理解这个项目

> 写给刚学编程的人看。不讲术语，只讲"是什么"和"为什么"。

---

## 一、这个项目是干什么的

温州有一个纺织厂，生产各种布条（衣服上的斜条、缎带、包边条）。下游的服装厂需要向这个工厂买布条。

以前是打电话下单，现在改成**微信小程序下单**——服装厂老板在手机上选商品、填地址、提交订单，工厂这边在手机上就能看到新订单、安排送货、处理退换货。

这个小程序就是干这件事的。

---

## 二、项目的两个部分

你可以把项目想象成一个餐厅：

| 餐厅的样子 | 对应到项目 | 用什么写的 |
|-----------|-----------|-----------|
| 顾客看到的菜单、点菜页面 | **小程序前端**（pages/ 文件夹） | 微信专用语言 |
| 厨房、厨师、账本 | **后端**（cloudfunctions/ 文件夹） | JavaScript（一种编程语言） |

### 前端（顾客能看到的）

就是小程序里的各种页面：
- **首页**：看所有商品，选颜色选尺寸，加购物车
- **购物车**：看看选了啥，删掉不要的
- **结算页**：填收货地址、手机号、提交订单
- **我的订单**：看自己下过的单，取消订单，申请退货换货
- **后台**（工厂员工用的）：管理订单、修改价格、处理退款

### 后端（顾客看不到的）

就是处理数据的地方，在微信的云服务器上运行：

- 顾客点"提交订单" → 数据存到数据库
- 工厂改订单状态 → 更新数据库 + 发微信通知给顾客
- 顾客退货 → 检查规则 → 更新数据

## 三、文件放在哪里（怎么看目录）

```
小程序项目/
├── pages/           ← 前端页面（你能在手机上看到的每一个画面）
│   ├── index/       ← 首页
│   ├── cart/        ← 购物车
│   ├── checkout/    ← 结算页
│   ├── orders/      ← 我的订单
│   └── admin/       ← 工厂后台（只有工厂的人能进）
│
├── cloudfunctions/  ← 后端（处理数据的"厨房"）
│   ├── submitOrder/ ← 处理"下单"这件事
│   ├── cancelOrder/ ← 处理"取消订单"这件事
│   └── ...
│
├── utils/           ← 工具文件（被很多页面共用的小功能）
├── docs/            ← 说明文档（你现在看的就是）
├── images/          ← 图片（图标什么的）
│
├── app.js           ← 小程序启动时的入口文件
├── app.json         ← 注册所有页面（告诉微信"我有这些页面"）
└── README.md        ← 项目说明（怎么安装、怎么运行）
```

**记住一个规律**：
- 每个页面是一个文件夹，里面有 4 个文件（.js 逻辑、.json 配置、.wxml 画面、.wxss 样式）
- 每个云函数也是一个文件夹，里面有 3 个文件（index.js 逻辑、package.json 依赖、config.json 超时）

## 四、后端代码的三层结构（最重要的一节）

每个云函数（处理数据的文件）应该分成三块，各管各的：

### 第 1 层：门口接待员（接口层）

**位置**：`exports.main` 这行开始的代码

**干什么**：
- 收下顾客的小纸条（就是前端传过来的数据）
- 检查小纸条上有没有填名字、电话（参数校验）
- 问一下"你是顾客还是工厂员工？"（权限校验）
- 然后把活交给后厨（业务层）
- 后厨做完后，把结果告诉顾客

**举例**（取消订单）：
```
接待员收到纸条："帮我取消订单号123"
→ 检查纸条没写错
→ 确认是顾客本人发的
→ 把活交给后厨
→ 后厨回来说"取消了"
→ 接待员告诉顾客："好了，取消了"
```

### 第 2 层：后厨（业务层）

**位置**：`doCancelOrder` 这种名字的函数

**干什么**：
- 不管纸条怎么来的，只管手上的活怎么做
- 比如"取消订单"这个活：
  - 先查一下这个订单存在吗
  - 这个订单是你的吗
  - 订单已经完成了吗（完成了就不能取消了）
  - 都没问题 → 把订单状态改成"已取消"

**重要**：后厨只管做菜，不管端盘子。它不直接和顾客说话。

### 第 3 层：仓库（数据层）

**位置**：`db.collection('订单表').doc('订单号').update(...)` 这种代码

**干什么**：
- 就是微信提供的数据库，像一个电子表格
- 不需要自己写，微信已经做好了
- 直接告诉它"把这一行的状态改成已取消"

## 五、接口返回的规矩（数据怎么回答）

每次后端处理完事情，要统一回答。就像餐厅的服务员，不管顾客点什么，回答的方式都一样的：

### 成功的时候

| 场景 | 怎么回答 | 例子 |
|------|---------|------|
| 给顾客看一堆东西（比如商品列表） | `res.list(东西的数组, 总共多少件)` | 返回 20 个商品，一共 1250 个 |
| 给顾客看一个东西（比如订单详情） | `res.record(这一个东西)` | 返回订单号 123 的信息 |
| 做完了但不返回东西（比如删除） | `res.ok()` | 删好了 |

### 失败的时候

| 场景 | 怎么回答 | 顾客看到什么 |
|------|---------|------------|
| 顾客没填手机号 | `res.badRequest('请填写手机号')` | 提示"请填写手机号" |
| 没登录 | `res.unauthorized()` | 提示"请先登录" |
| 登录了但不是管理员 | `res.forbidden()` | 提示"无权限" |
| 查的东西不存在 | `res.notFound('订单不存在')` | 提示"订单不存在" |
| 操作不允许（比如已完成不能取消） | `res.conflict('订单已完成，无法取消')` | 提示不能取消的原因 |
| 服务器自己出错了 | `res.internalError()` | 提示"操作失败，请稍后重试" |

**为什么搞这么多种回答？** 因为不同的错，前端可以有不同的反应。比如"未登录"就跳转登录页，"无权限"就提示不能看，"参数错误"就在输入框旁边标红。

## 六、错误处理的规矩（出错了怎么办）

### 三层保护

```
第 1 层：每个云函数里面 → try/catch 包住所有代码
          ↓ 出错了
第 2 层：记一笔"什么错了、哪个订单出的错" → logger.error(...)  
          ↓
第 3 层：告诉顾客"操作失败，请稍后重试" → res.internalError()
```

**为什么不能把真实的错误信息告诉顾客？** 

比如数据库出错了，真实信息可能是 `Cannot read property 'data' of undefined`，顾客看到这种英文完全不知道什么意思。所以告诉顾客一个友好的中文提示就行了，真实错误记录在日志里给程序员看。

---

## 七、日志的规矩（记笔记）

就像餐厅有个记事本，每件重要的事都要记一笔：

| 级别 | 什么时候记 | 例子 |
|------|-----------|------|
| **info**（普通） | 正常完成了一件事 | "顾客张三下了订单 123，金额 250 元" |
| **warn**（警告） | 出了小问题但不影响主要流程 | "给顾客发微信通知失败了" |
| **error**（错误） | 出了大问题需要修 | "数据库写入失败" |

用 `logger.info('事情名字', { 订单号: '123' })` 这种写法记。

**为什么要记日志？** 出了问题时，翻日志就能知道"是哪个订单、什么时候、出了什么错"。不然就像蒙着眼睛找东西。

## 八、权限的规矩（谁能干什么）

### 两种身份

| 身份 | 怎么判断 | 能干什么 |
|------|---------|---------|
| **顾客** | 微信自动给每个人一个编号（openid） | 看自己的订单、下单、取消自己的订单 |
| **工厂员工** | 在数据库的"管理员表"里有记录 | 看所有订单、改价格、处理退货 |

### 怎么检查身份

不要在每个云函数里重复写检查代码。用一个公共模块 `auth.js`：

```js
// 检查是不是顾客
const result = await auth.requireOpenid();
if (!result.authorized) return result.response;  // 不是顾客，拒绝

// 检查是不是工厂员工
const result = await auth.requireAdmin();
if (!result.authorized) return result.response;  // 不是员工，拒绝
```

**为什么要统一？** 如果 15 个云函数各自写检查方式，哪天要改检查规则，就得改 15 个文件。统一到一个文件，只改一处。

## 九、数据库的规矩（数据怎么存）

### 几条铁律

1. **价格用"分"存，不用"元"**
   - 比如 1.5 元 → 存 `150`（150 分）
   - 为什么？电脑算小数会出错。`0.1 + 0.2` 不等于 `0.3`，而是 `0.3000000000004`。用整数（分）就不会有这个问题。

2. **时间用服务器时间，不用手机时间**
   - 写 `db.serverDate()` 而不是 `new Date()`
   - 为什么？顾客的手机时间可能不准（慢了 3 小时之类的），服务器时间是准的。

3. **查东西要分页，不能一次全拿出来**
   - 每页最多 100 条
   - 为什么？如果有 10000 条数据一次全拿出来，会卡死。

4. **顾客的数据必须打上标记**
   - 存订单时必须加上 `_openid: '顾客的微信编号'`
   - 查订单时必须加上条件 `_openid: '当前顾客的编号'`
   - 为什么？防止顾客 A 看到顾客 B 的订单。

## 十、怎么添加一个新功能

### 如果要加一个新页面（前端）

1. 在 `pages/` 下建一个文件夹
2. 文件夹里建 4 个文件：
   - `xxx.js` — 页面逻辑
   - `xxx.wxml` — 页面样子
   - `xxx.wxss` — 页面颜色大小
   - `xxx.json` — 页面配置
3. 在 `app.json` 的 `pages` 列表里加上这个页面路径
4. 如果要做成底部标签页，在 `app.json` 的 `tabBar` 里也加上

### 如果要加一个新云函数（后端）

1. 在 `cloudfunctions/` 下建一个文件夹
2. 文件夹里建 3 个文件：
   - `index.js` — 处理逻辑（用三层的写法）
   - `package.json` — 声明依赖（写 `"wx-server-sdk": "~2.6.3"`）
   - `config.json` — 超时时间（写 `"timeout": 30`）
3. 在 `cloudbaserc.json` 的 `functions` 列表里加上这个云函数
4. 部署：`tcb fn deploy 函数名 --force`

### js 文件的模板（复制粘贴，改函数名就行）

```js
const cloud = require('wx-server-sdk');
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV });
const db = cloud.database();
const res = require('../lib/response');
const logger = require('../lib/logger');
const auth = require('../lib/auth');

exports.main = async (event) => {
  try {
    // 第1步：检查参数
    if (!event.必填参数) return res.badRequest('请填写必填参数');

    // 第2步：检查身份（二选一）
    const authResult = await auth.requireOpenid();  // 顾客
    // const authResult = await auth.requireAdmin(); // 管理员
    if (!authResult.authorized) return authResult.response;

    // 第3步：做事（查数据库、判断规则）
    const result = await db.collection('表名').xxx(数据);

    // 第4步：记日志 + 返回
    logger.info('做了什么事', { 关键信息 });
    return res.record(结果数据);  // 或 res.ok() 或 res.list()

  } catch (err) {
    logger.error('函数名', err, { 入参关键信息 });
    return res.internalError();
  }
};
```

## 十一、常见问题（真的会遇到）

### 为什么部署后功能没用？
1. 云函数有没有部署上去？运行 `tcb fn list` 看看在不在
2. `config.json` 的 `timeout` 写了没？默认只有 3 秒，大概率不够
3. `package.json` 的依赖写了没？缺少依赖部署会失败
4. 数据库集合创建了没？没创建的话存不了数据

### 为什么代码在电脑上能跑，手机上不行？
1. 是不是用的测试号？测试号没有云开发权限，要用正式 AppID
2. 云函数是不是只部署了部分？改了哪个就得重新部署哪个
3. 小程序是不是没通过审核？没审核的小程序只有你自己能看到

### 为什么以前能用的功能突然不能用了？
1. 是不是改了代码没重新部署？
2. 是不是云平台欠费了？
3. 打开微信开发者工具的控制台，看看有没有红色报错

## 十二、这个词是什么意思（速查表）

| 词 | 大白话解释 |
|----|-----------|
| 云函数 | 在微信服务器上跑的一段代码。你不用管服务器在哪、怎么配置，写好了传上去就能用 |
| 数据库 | 电子表格。存产品、订单、用户信息的地方 |
| 集合 | 数据库里的一张表。比如"产品表"存所有产品，"订单表"存所有订单 |
| openid | 微信给每个用户的一个编号。不是你起的名字，是微信自动给的。用来区分"这个订单是谁的" |
| SDK | 别人写好的工具包。微信把常用的功能（存数据、发通知）都打包好了，你直接调用就行，不用自己写 |
| 部署 | 把写好的代码上传到服务器上让它跑起来 |
| 鉴权 | 检查"你是谁、你有资格吗" |
| 分页 | 每次只拿一部分数据（比如一次拿 20 条），避免一次拿太多卡死 |
| logger | 日志。就是自动记笔记的东西 |
| config.json | 配置文件。告诉微信"我这个函数最多等 30 秒，超过就别等了" |
| package.json | 依赖清单。告诉微信"我需要用这些工具包" |
| try/catch | 保护代码。try 里面正常做事，如果出错了 catch 能兜住，不会整个程序崩溃 |

## 十三、为什么代码不能堆在一起

这一节解释四个"为什么"。不是背规则，是理解原因。理解了原因，自然就知道怎么写了。

---

### 13.1 为什么请求入口不能堆复杂业务判断

**"请求入口"** 就是 `exports.main` 这个函数——每次前端调用云函数，第一个被执行的就是它。

**堆在一起的坏处**（用取消订单举例）：

```js
// ❌ 全部堆在 exports.main 里——看着就头晕
exports.main = async (event) => {
  const { orderId } = event;
  const wxContext = cloud.getWXContext();
  const openid = wxContext.OPENID;
  if (!orderId) return { code: -1, msg: '参数错误' };
  if (!openid) return { code: -1, msg: '未登录' };
  const order = await db.collection('orders').doc(orderId).get();
  if (!order.data) return { code: -1, msg: '订单不存在' };
  if (order.data._openid !== openid) return { code: -1, msg: '无权操作' };
  if (order.data.status === 'completed') {
    return { code: -1, msg: '订单已完成，无法取消' };
  }
  if (order.data.status === 'cancelled') return { code: -1, msg: '订单已取消' };
  if (order.data.returnRequest) return { code: -1, msg: '已有退换货申请' };
  await db.collection('orders').doc(orderId).update({
    data: { status: 'cancelled', updatedAt: db.serverDate() }
  });
  return { code: 0 };
};
```

**这段代码的问题**：

1. **读不懂** — 一眼看过去，接收参数、检查身份、验证规则、读写数据库全挤在一起。20 行还能忍，如果业务规则有 50 行呢？80 行呢？

2. **改不动** — 假如要加一条规则"VIP 客户的订单不能自己取消，要打电话给厂长取消"，你得在 exports.main 中间找地方插进去。找着找着就改错了。

3. **不能复用** — 假设另一个云函数也要判断"订单存不存在、是不是你的"，你只能把这一段复制粘贴过去。以后改规则就得改两个地方。

**拆开后**：

```js
// ✅ 拆开——入口只做"接待"，具体活交给下面的人
exports.main = async (event) => {
  try {
    const v = validate(event);                          // 检查参数
    if (!v.valid) return res.badRequest(v.error);

    const authResult = await auth.requireOpenid();      // 检查身份
    if (!authResult.authorized) return authResult.response;

    const bizResult = await doCancelOrder(v.orderId, authResult.openid);  // 做事
    if (!bizResult.success) return res.conflict(bizResult.error);

    return res.ok();
  } catch (err) {
    logger.error('cancelOrder', err, { orderId: event.orderId });
    return res.internalError();
  }
};
```

**好处**：
- exports.main 只做"接待和调度"——叫谁干活、检查结果、返回答案。
- 要改业务规则？去 `doCancelOrder` 改，不用碰 exports.main。
- `auth.requireOpenid()` 被所有云函数共用，改一处全项目生效。
- 任何一个程序员（包括三个月后的你）打开这个文件，5 秒就能看懂结构。

**一句话总结**：入口函数就像前台，只负责"您找谁、稍等、好了请慢走"。具体活是后厨干的。

---

### 13.2 为什么数据库访问不能散落在各个请求函数里

先解释"散落"是什么意思——假设每个云函数都这样写：

```js
// adminGetOrders 里
const orders = await db.collection('orders').where({ status: 'processing' }).get();

// submitOrder 里
const result = await db.collection('orders').add({ data: orderData });

// cancelOrder 里
await db.collection('orders').doc(orderId).update({ data: { status: 'cancelled' } });
```

每个函数都直接操作数据库，看着没问题？**问题在以后**。

**场景一：加一个字段**。比如要在每个订单上加 `shopId`（哪个仓库的订单）。你需要找到所有操作 `orders` 表的地方——3 个、5 个、10 个云函数——逐一修改。漏一个就出 bug。

**场景二：改权限规则**。比如改为"顾客只能看自己最近 90 天的订单"。如果每个函数都自己写 `where({ _openid: openid })`，你得挨个加 `where({ createdAt: ... })`。

**场景三：换数据库**（虽然可能性小）。如果哪天 CloudBase 涨价了要换别的服务，所有散落的 `db.collection()` 调用都要改。

**正确做法是**：数据库操作放在业务函数 `doXxx()` 里，只在它需要的地方出现。如果同样的查询出现在 3 个以上云函数，就抽出来放到 `lib/` 下。

**不是说不让直接用 `db.collection()`**——CloudBase 的 `db` 对象本身就是数据层，不需要再包一层。但"直接调用"不等于"到处乱写"。应该有规律：
- 同一个表的同类查询，集中在同一个函数里
- 如果会被多个地方用到，放 `lib/` 下

**一句话总结**：数据库操作就像用电——每个房间都有插座（都可以直接调 db），但电线走线要有规划（查询要集中，不散落）。不然哪天线路出问题，你不知道该修哪个插座。

### 13.3 为什么配置不能写死在业务代码里

**"写死"** 是什么意思？就是在代码里直接写数字、字符串，而不是放到配置文件里。

**举个例子**：

```js
// ❌ 写死的
async function doLogin(username, password) {
  const result = await db.collection('admins').where({ username }).get();
  if (result.data[0].failedAttempts >= 5) {   // ← "5" 是什么？为什么是 5 不是 3？
    const lockedUntil = result.data[0].lockedUntil;
    if (lockedUntil > Date.now() - 15 * 60000) {  // ← "15" 又是什么？
      return { success: false, error: '账号已锁定' };
    }
  }
}
```

一个月后你回来看这段代码，完全不记得 5 和 15 是什么意思。要改的话还得在代码中间找。

**正确做法**：

```js
// ✅ 用常量
const MAX_ATTEMPTS = 5;       // 最多输错 5 次
const LOCK_MINUTES = 15;      // 锁定 15 分钟

async function doLogin(username, password) {
  if (result.data[0].failedAttempts >= MAX_ATTEMPTS) {
    if (lockedUntil > Date.now() - LOCK_MINUTES * 60000) {
      return { success: false, error: '账号已锁定' };
    }
  }
}
```

**好处**：
- 一眼知道 `MAX_ATTEMPTS = 5` 是什么意思（"最多输错次数"）
- 要改成"输错 3 次就锁"，只改一行（`const MAX_ATTEMPTS = 3`），不用在代码中间翻找
- 如果多个地方用到这个值（比如登录页和后台页都显示"剩余尝试次数"），保证用的是同一个数

**哪些东西应该提到配置文件/常量里**：

| 写死的写法 | 应该改成 |
|-----------|---------|
| `if (price >= 10000)` | `const FREE_SHIPPING_THRESHOLD = 10000;` // 满多少免运费 |
| `setTimeout(..., 8000)` | `const PROBE_TIMEOUT = 8000;` // 探活超时 |
| `'cloudbase-d6g98vaoyb7ec331a'` | `const ENV_ID = 'cloudbase-d6g98vaoyb7ec331a';` 或单独配置文件 |
| `wx-server-sdk: "latest"` | `"~2.6.3"` — 锁定版本，防止新版本不兼容 |
| `pageSize: 20` （出现 5 次） | `const DEFAULT_PAGE_SIZE = 20;` |

**一句话总结**：代码里不应该有"魔法数字"（突然出现、你不知道它为什么是这个值的数字）。要么用有名字的常量代替，要么放到配置文件里。这样以后改的时候不用猜"这个数字是干什么的"。

### 13.4 总结：三层拆分的本质

```
exports.main（接待员）
  │  只管：收参数 → 叫校验员 → 叫保安 → 叫后厨 → 告诉结果
  │  不管：这个订单能不能取消、金额怎么算
  │
  ├─ validate()（校验员）
  │    只管：名字填了吗、手机号格式对吗、数量大于 0 吗
  │    不管：这个人是不是会员、订单是谁的
  │
  ├─ auth.requireXxx()（保安）
  │    只管：你是谁、你有权限吗
  │    不管：你要干什么、数据怎么改
  │
  └─ doXxx()（后厨）
       只管：查数据、判断业务规则、写入数据库
       不管：参数从哪来的、是谁在操作、结果怎么告诉顾客
```

**为什么要这样分？因为改需求时，通常只动其中一层**：

- 加校验规则 → 只改 `validate()`
- 改权限 → 只改 `lib/auth.js`
- 改业务规则（比如"VIP 不能自己取消订单"）→ 只改 `doCancelOrder()`
- 改返回格式 → 只改 `lib/response.js`

不分层的话，改一行需求可能要动 5 个云函数、10 处代码。分好层的话，可能只改一个函数。

**你现在可能觉得"我的代码才 20 行，分层太麻烦了"。但 20 行会变成 80 行，80 行会变成 200 行。早分层，比以后重构要省 10 倍的时间。**
